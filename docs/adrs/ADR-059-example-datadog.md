# ADR-059: example-datadog — Datadog SDK showcase

**Status**: Proposed
**Date**: 2026-06-17
**Project**: `ruvnet/agent-harness-generator`
**Related**: ADR-051 (examples program), ADR-022 (MCP default-deny), ADR-026 (tiered routing), ADR-050 (verification-gated output)

---

## Context

The MetaHarness examples program (ADR-051) ships one self-contained, npx-runnable package per third-party SDK. This record covers the Datadog entry.

Datadog is the dominant commercial observability platform. Its TypeScript client (`@datadog/datadog-api-client`, currently v1.57.0) exposes REST APIs through two versioned namespaces:

- **v1** – monitors, dashboards, events, legacy metrics (`v1.MetricsApi`, `v1.MonitorsApi`)
- **v2** – logs search, timeseries queries, incidents, tags (`v2.MetricsApi`, `v2.LogsApi`, `v2.IncidentsApi`)

Authentication follows a two-key model: `DD_API_KEY` (reporting access) plus `DD_APP_KEY` (read/management access). The target site/region is set via `DD_SITE` (default `datadoghq.com`; EU: `datadoghq.eu`; AP1: `ap1.datadoghq.com`; US3: `us3.datadoghq.com`; US5: `us5.datadoghq.com`; Gov: `ddog-gov.com`). The SDK picks up all three values automatically from the environment.

The headline use-case is **incident triage**: given a firing alert, automatically correlate related metric anomalies, relevant log bursts, and the state of associated monitors, then produce a structured root-cause hypothesis — all without mutating any Datadog resource.

The scaffold must run on all nine MetaHarness hosts (claude-code, codex, copilot, github-actions, hermes, openclaw, opencode, pi-dev, rvm) via the `--host` flag.

## Decision

### Chosen SDK

`@datadog/datadog-api-client` (npm, Datadog-maintained, TypeScript-first). Selected because it is the official Datadog-published monorepo client, covers all relevant v1/v2 API namespaces in a single install, and follows the same env-var convention (`DD_API_KEY` / `DD_APP_KEY` / `DD_SITE`) that operators already use for the Datadog Agent.

### Headline read capabilities demonstrated

| Capability | SDK class | Namespace |
|---|---|---|
| Query timeseries metrics | `v2.MetricsApi.queryTimeseries()` | v2 |
| Search logs | `v2.LogsApi.listLogs()` | v2 |
| List / get monitors | `v1.MonitorsApi.listMonitors()` | v1 |
| List incidents | `v2.IncidentsApi.listIncidents()` (`unstableOperations["v2.listIncidents"] = true`) | v2 |

### Agents and /slash command

- **dd-collector** (planner tier) — receives a triage prompt, fans out parallel read calls for metrics, logs, and monitors, and assembles raw evidence into a structured payload.
- **dd-analyst** (reasoning tier) — consumes the evidence payload and synthesises a ranked root-cause hypothesis with confidence scores.
- **dd-verifier** (verification gate, ADR-050) — re-queries the primary metric and the referenced monitor immediately before finalising output; if the re-query contradicts the hypothesis, the analyst is re-invoked with the updated evidence before the response is surfaced.

**`/dd-triage`** — entry-point slash command. Accepts a natural-language description of a firing alert and optional `--service`, `--env`, and `--from`/`--to` time-range flags. Orchestrates dd-collector → dd-analyst → dd-verifier in sequence.

### Tiered routing (ADR-026)

| Tier | Model | Agents | Rationale |
|---|---|---|---|
| 1 | Haiku | dd-collector | High-volume, low-complexity fan-out: parallel REST reads + JSON extraction. Latency-sensitive; cheap. |
| 2 | Sonnet / Opus | dd-analyst, dd-verifier | Root-cause reasoning and contradiction detection require deeper inference. |

Routing is expressed in `.harness/routing.json` generated by the scaffold; the MetaHarness CLI respects it via `@metaharness/host-<id>` host adapters.

### MCP policy (ADR-022 default-deny)

The scaffold writes `.harness/mcp-policy.json` granting exactly the following read-only tools:

```json
{
  "default": "deny",
  "grants": [
    "datadog:metrics.queryTimeseries",
    "datadog:logs.listLogs",
    "datadog:monitors.listMonitors",
    "datadog:monitors.getMonitor",
    "datadog:incidents.listIncidents",
    "datadog:incidents.getIncident",
    "datadog:auditLogs.listAuditLogs"
  ]
}
```

All mutation tools (`datadog:monitors.muteMonitor`, `datadog:incidents.createIncident`, `datadog:incidents.updateIncident`, `datadog:events.createEvent`) are **absent** from the grants list and therefore denied by default.

### Auth model

Secrets are consumed exclusively from environment variables:

```
DD_API_KEY   – Datadog API key (required)
DD_APP_KEY   – Datadog application key (required for reads)
DD_SITE      – Datadog site region (optional, default: datadoghq.com)
```

The scaffold generates a `.env.example` with placeholders. No secret is written to a source file or committed (a generated `.gitignore` entry covers `.env`). Configuration is initialised at runtime:

```typescript
import { client, v1, v2 } from '@datadog/datadog-api-client';
const configuration = client.createConfiguration(); // reads DD_* from process.env
```

### Safety gates

- **Read-only by default**: no mutation MCP grant is included in the default policy.
- **Opt-in mutations**: operators may pass `--allow-mutations` to the scaffold CLI, which appends a clearly labeled mutation grant block to `.harness/mcp-policy.json` and emits a conspicuous warning in the generated project's README.
- **Verification gate (ADR-050)**: dd-verifier re-queries before output; a mismatch triggers a re-analysis loop (max 2 iterations) before surfacing results.
- **No hardcoded credentials**: enforced at scaffold generation time by linting the emitted files.
- **`unstableOperations` flag**: `v2.listIncidents` is marked unstable by Datadog; the scaffold enables it explicitly and logs a warning at startup so operators are aware of the stability posture.

### Host wiring

The scaffold calls `metaharness scaffold --host all` (or `--host <id>`) which delegates to the nine `@metaharness/host-*` adapters. Each host adapter translates `.harness/routing.json` and `.harness/mcp-policy.json` into host-native configuration.

## Consequences

### Positive

- Operators get a working, npx-runnable incident-triage bot in minutes with zero secret exposure.
- Tiered routing keeps Haiku costs low for the high-frequency collector fan-out while reserving Sonnet/Opus tokens for reasoning.
- MCP default-deny prevents any accidental mutation of production monitors or incidents during exploratory use.
- The verification gate catches transient metric spikes that resolve between the collector call and the analyst synthesis, reducing false-positive root-cause hypotheses.
- Covers all nine MetaHarness hosts via a single `--host all` flag.

### Honest limits

- `v2.listIncidents` is flagged `unstableOperations` by Datadog; the API contract may change across minor SDK versions.
- Timeseries queries are bounded by Datadog's default query window and rate limits; high-cardinality queries may be throttled.
- Log search returns paginated results; the collector fetches one page (up to 1 000 events) by default — deep log archaeology requires additional pagination.
- The Incidents API requires the **Incident Management** feature on the org; legacy plans may receive 403 errors.
- `DD_SITE` must be set correctly for EU/AP/Gov customers; the scaffold emits a `doctor` check that validates site reachability before any agent runs.

### Not for production

This package is an **illustrative MetaHarness showcase**, not a production incident-management system. Output is AI-generated analysis subject to human review. Do not use it as the sole basis for production change decisions.
